# Git

```bash
sudo pacman -S --needed git git-lfs
yay -S --needed git-credential-manager-bin
git lfs install

git config --global init.defaultBranch main
git config --global core.autocrlf input
git config --global core.quotepath false
git config --global core.editor "nvim --clean"
git config --global user.name "your-name"          # edit name
git config --global user.email "you@example.com"   # edit email

# diff/merge: 使用 git-delta 和 neovim
git config --global core.pager delta
git config --global interactive.diffFilter "delta --color-only"
git config --global delta.navigate true
git config --global delta.hyperlinks true
git config --global delta.side-by-side true
git config --global delta.line-numbers true
git config --global delta.syntax-theme base16
git config --global diff.tool nvimdiff
git config --global difftool.prompt false
git config --global merge.tool nvimdiff
git config --global mergetool.prompt false
git config --global merge.conflictStyle zdiff3

git config --global credential.credentialStore secretservice
git-credential-manager configure # 注册 GCM 为 credential.helper
# git-credential-manager unconfigure # 取消注册

# 已在.bashrc中自动配置
# git config --global http.proxy "http://127.0.0.1:10808"
# git config --global https.proxy "http://127.0.0.1:10808"

# 可选择给GitHub添加SSH认证
# ssh-keygen -t ed25519 -C "you@example.com" # 生成ED25519类型的SSH密钥对, 用邮箱标识
# eval "$(ssh-agent -s)"                     # 启动SSH代理, 用于管理SSH密钥
# ssh-add ~/.ssh/id_ed25519                  # 将生成私钥添加到SSH代理
# cat ~/.ssh/id_ed25519.pub                  # 查看公钥
# 访问https://github.com/settings/keys, 点击"New SSH key", 粘贴公钥并保存
# ssh -T git@github.com            # 测试SSH连接是否正常

# 个别项目常见可选配置:
# git config pull.rebase true
# git config rebase.autoStash true
# git config rerere.enabled true
```

# 开发环境工具

```bash
curl https://mise.run | sh # 原生安装, 不用包管理器
mise trust --all ~/.config/mise # 信任stow软链接, 使配置生效

# dotfiles(stow) 涉及配置:
# 1. mise全局配置: ~/.config/mise/config.toml , 包括需要安装工具的清单和配置
# 2. shell激活配置(.bashrc 和 fish config)

mise self-update # 更新mise
mise upgrade # 安装/升级工具

# 安装工具命令示例(无需执行)
# mise install java@temurin-8 java@25 maven@3 node@lts uv@latest
# mise use -g java@25 maven@3 node@lts uv@latest
```

---

在命令行用 mise 下载更新常常 github 限流,  
需要去 [Github Token](https://github.com/settings/tokens) 获取 token(不需要添加权限),  
然后导入`GITHUB_TOKEN`环境变量.  
可以放`~/.config/environment.d/private.conf`, 格式`GITHUB_TOKEN=xxx`,  
也可以 Fish Shell 中执行 `set -gx GITHUB_TOKEN xxx`.

---

mise 支持安装 npm 的工具(用`npm:`前缀),  
用 `mise use -g` 且安装了 aube, mise 会自动用 aube 作为包管理工具

**全面使用 aube 代替 npm**:

| 场景         | npm 原用法           | 代替用法              |
| ------------ | -------------------- | --------------------- |
| 全局安装工具 | `npm install -g`     | `mise use -g npm:xxx` |
| 执行临时包   | `npx xxx`            | `aubx xxx`            |
| 项目安装依赖 | `npm install`        | `aube install`        |
| 添加依赖     | `npm install lodash` | `aube add lodash`     |
| 运行脚本     | `npm run build`      | `aubr build`          |

npm 即使在项目内执行也会污染全局:

- `npm install` 会写缓存到 `~/.npm/_cacache/`, 删除项目后缓存仍残留
- `npx` 会写到 `~/.npm/_npx/` 和 `~/.npm/_cacache/`
- npm 所有依赖的 lifecycle 脚本 (preinstall/install/postinstall) 默认全部以用户权限执行, 存在安全风险

aube 的优势:

- 全局存储遵循 XDG 规范: `~/.local/share/aube/store/`
- 依赖 lifecycle 脚本默认不执行, 需要 `aube approve-builds` 显式批准
- 项目目录更干净: 隔离 node_modules 布局, 防止幽灵依赖

aube 维护命令:

```bash
aube store status  # 查看全局存储状态 (大小、文件数)
aube store prune   # 清理未被任何项目引用的缓存文件
aube store path    # 查看全局存储路径
```

aubx 常用参数:

```bash
aubx cowsay hello                         # 执行临时包
aubx -p @anthropic-ai/claude-code         # -p: 包名与二进制名不同时指定包名
aubx -p prettier@3 -- prettier --check .  # 指定包名+版本, -- 后是传给二进制的参数
aubx -c 'cowsay hi | tr a-z A-Z'          # -c: shell 模式, 支持管道
```

---

mise 可管理 python 版本, 但系统自带了 python, 且已经通过mise安装了 `uv`,  
因此不使用 mise 下载和管理 python, 都用 uv.  
mise 只用于切换到python项目目录时自动激活已存在的 python 虚拟环境(不介入依赖安装)

python(uv) 项目中的 `mise.toml` 举例:

```toml
# need to execute mise trust
[env]
PROJECT_NAME = "{{ config_root | basename }}"
_.python.venv = { path = ".venv", create = false }
[settings]
python.uv_venv_auto = "source"
```

Mise 常用命令
| 操作           | 命令                               | 说明                          |
| -------------- | ---------------------------------- | ----------------------------- |
| 搜索工具       | `mise registry [关键词]`           | 查看支持哪些工具              |
| 查看可用版本   | `mise ls-remote node`              | 远程版本列表                  |
| 仅安装         | `mise install node@lts`            | 安装但不激活                  |
| 安装并激活     | `mise use -g node@lts`             | 安装 + 写入全局配置           |
| 查看已安装     | `mise ls`                          | 列出所有工具及版本            |
| 查看过期       | `mise ls --outdated`               | 查看可升级的工具              |
| 升级工具       | `mise upgrade`                     | 升级所有工具                  |
| 交互式升级     | `mise upgrade --interactive`       | 选择性升级                    |
| 升级+改版本号  | `mise upgrade --bump`              | 同时更新 mise.toml 中的版本号 |
| 卸载工具       | `mise uninstall node@20`           | 移除特定版本                  |
| 移除激活       | `mise use --global --remove node`  | 从配置中移除                  |
| 查看安装路径   | `mise where node@lts`              | 工具安装目录                  |
| 临时执行       | `mise exec node@22 -- node app.js` | 用指定版本一次性运行          |
| 更新 mise 自身 | `mise self-update`                 | 更新 mise CLI                 |

Code CLI的环境管理规范

可复制到 `~/.claude/CLAUDE.md` 之类的文件

```md
- Markdown 多行连续句子用两个空格换行
- Python 统一用 uv (不用原生 pip/venv/requirements.txt):
  - 创建项目: `uv init`
  - 添加/移除依赖: `uv add pkg` / `uv remove pkg`
  - 安装依赖: `uv sync`
  - 运行命令: `uv run python app.py`
  - 临时执行带依赖的脚本: `uv run --with pkg script.py`
  - 执行临时工具: `uvx tool` (类似 pipx)
  - 已有 requirements.txt 的项目: `uv venv && uv pip install -r requirements.txt`
  - 已有 pyproject.toml 的项目: `uv sync`
  - 不要 `pip install`, 不要 `python -m venv`
- Node.js 统一用 aube (不用 npm/npx):
  - 安装依赖: `aube install`
  - 添加依赖: `aube add pkg`
  - 添加开发依赖: `aube add -D pkg`
  - 运行脚本: `aubr script_name`
  - 执行临时包: `aubx pkg`
  - 不要 `npm install`, 不要 `npx`
- 全局工具用 `mise use -g pkg` 或 `mise use -g npm:pkg` 安装, 禁止 `npm install -g`
- mise 管理开发相关工具: 编程语言/版本依赖管理, code CLI, 全局 npm 包
- 系统整洁: 遵循 XDG 规范, 不在 $HOME 创建散落的 dotfiles
```

网络功能测试

```bash
# netshoot: 自带大量网络调试命令的容器镜像
docker run --rm -it nicolaka/netshoot
# --rm: 容器退出后删除 ; --it: 保持stdin输入+分配终端=进入交互式shell
# 命令通过--network参数指定各种网络模式, 比如: --network host
# 查看容器网络模式: docker inspect [容器名称] | grep NetworkMode

# ip a                           # 查看ip配置
# ip route                       # 查看路由配置
# ping 1.1.1.1                   # 测试ip连接
# dig google.com                 # 测试DNS

# ===代理(docker默认网络模式)===
# 1. 系统代理
# 代理软件要配置"允许局域网的连接"(监听0.0.0.0)
# docker run 命令增加 --add-host=host.docker.internal:host-gateway 参数, 用于访问宿主机
# env | grep -i proxy # 查看容器内代理环境变量
```

---

**Docker Compose 配置注意项**:

Docker Compose 自定义网络默认从 172.18.0.0/16 开始分配,  
与 singbox TUN 默认网段 (172.18.0.1/30) 可能冲突.  
冲突时，访问 Docker 容器服务的流量会被错误路由到 TUN 接口.

检查: `ip route show | grep 172`

解决方法:

1. docker-compose.yml 指定网段(最简单，不影响其他项目)
2. Docker daemon 配置默认网段池(全局生效)
3. 修改 singbox TUN 网段(需要客户端支持)

方法1示例:

```yml
networks:
  my-network-name:
    driver: bridge
    ipam:
      config:
        - subnet: 172.28.0.0/16
          gateway: 172.28.0.1
```
